Bug Title / Short Description (one liner) : "Our site gets slower every day — and we can't figure out why"
Bug category: Core Settings & Options API



Description: 
A WooCommerce store owner reported that their site had been progressively slowing down over three months — from a 1.2 second page load at launch to over 6 seconds, with no changes to their content or traffic. A database size check revealed wp_options had grown from 2MB at launch to over 180MB — and was growing by approximately 2MB every day.





Main Cause:
A combination of two plugins was responsible for the runaway options growth:

A WooCommerce analytics plugin was storing per-session visitor data as individual wp_options rows — one row per site visit — with option names like wc_session_abc123def456. After three months of ~800 daily visitors, this had created over 70,000 rows of session data, most of it expired but never cleaned up.
A form builder plugin was storing each form submission as a serialized array in wp_options with autoload = yes — meaning all 40,000+ submissions were being loaded into PHP memory on every single page request, regardless of whether forms were displayed on that page.







What We Did:


Step 1 — Diagnosed the bloat with a precise options audit:

# Find the biggest options by storage consumed

SELECT
    option_name,
    LENGTH(option_value)                              AS bytes,
    ROUND(LENGTH(option_value) / 1024, 1)            AS kb,
    autoload,
    SUBSTRING(option_value, 1, 80)                   AS preview
FROM wp_options
ORDER BY bytes DESC
LIMIT 30;




# Find groups of options by name prefix (reveals plugin patterns)


SELECT
    SUBSTRING_INDEX(option_name, '_', 2)             AS name_prefix,
    COUNT(*)                                          AS row_count,
    ROUND(SUM(LENGTH(option_value)) / 1024 / 1024, 2) AS total_mb,
    SUM(CASE WHEN autoload = 'yes' THEN 1 ELSE 0 END) AS autoloaded_count
FROM wp_options
GROUP BY name_prefix
ORDER BY total_mb DESC
LIMIT 20;



-- Results:
-- wc_session     →  71,204 rows  →  94.3 MB  →  0 autoloaded
-- formbuilder    →  41,887 rows  →  68.1 MB  →  41,887 autoloaded  ← catastrophic







Step 2 — Cleaned expired WooCommerce session rows in safe batches:


// cleanup-wc-sessions.php — Run via: wp eval-file cleanup-wc-sessions.php

global $wpdb;

$batch_size    = 1000;
$total_deleted = 0;


// WC sessions store a timeout in a companion _timeout option
// Delete sessions where the companion timeout has already passed

do {
    $deleted = $wpdb->query("
        DELETE FROM {$wpdb->options}
        WHERE option_name LIKE 'wc_session_%'
        AND option_name NOT LIKE '%_timeout'
        AND option_name IN (
            SELECT REPLACE(t.option_name, '_timeout', '') FROM (
                SELECT option_name FROM {$wpdb->options}
                WHERE option_name LIKE 'wc_session_%_timeout'
                AND   option_value < UNIX_TIMESTAMP()
                LIMIT {$batch_size}
            ) AS t
        )
        LIMIT {$batch_size}
    ");

    // Also clean up orphaned timeout rows
    $wpdb->query("
        DELETE FROM {$wpdb->options}
        WHERE option_name LIKE 'wc_session_%_timeout'
        AND   option_value < UNIX_TIMESTAMP()
        LIMIT {$batch_size}
    ");

    $total_deleted += $deleted;
    echo "Batch complete — total deleted: $total_deleted\n";
    usleep(100000); // 100ms pause between batches
} while ($deleted > 0);

echo "Session cleanup complete. Total rows deleted: $total_deleted\n";
$wpdb->query("OPTIMIZE TABLE {$wpdb->options}");
echo "Table optimized.\n";








Step 3 — Migrated form submissions out of wp_options into a dedicated custom table:

// migrate-form-submissions.php — Move form data to a proper custom table

global $wpdb;
$table_name = $wpdb->prefix . 'form_submissions';


// Create dedicated submissions table
$wpdb->query("
    CREATE TABLE IF NOT EXISTS {$table_name} (
        id           BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
        form_id      VARCHAR(100)    NOT NULL,
        submitted_at DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
        form_data    LONGTEXT        NOT NULL,
        user_id      BIGINT UNSIGNED          DEFAULT NULL,
        ip_address   VARCHAR(45)              DEFAULT NULL,
        PRIMARY KEY (id),
        KEY idx_form_id      (form_id),
        KEY idx_submitted_at (submitted_at)
    ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
");


// Migrate existing form data from wp_options to new table
$form_options = $wpdb->get_results("
    SELECT option_name, option_value
    FROM {$wpdb->options}
    WHERE option_name LIKE 'formbuilder_submission_%'
    ORDER BY option_id ASC
");

$migrated = 0;
foreach ($form_options as $option) {
    $data = maybe_unserialize($option->option_value);
    if (!is_array($data)) continue;

    $wpdb->insert($table_name, [
        'form_id'      => sanitize_key($data['form_id'] ?? 'unknown'),
        'submitted_at' => $data['submitted_at'] ?? current_time('mysql'),
        'form_data'    => wp_json_encode($data),
        'user_id'      => !empty($data['user_id']) ? absint($data['user_id']) : null,
        'ip_address'   => sanitize_text_field($data['ip'] ?? ''),
    ]);
    $migrated++;
}
echo "Migrated $migrated submissions to dedicated table.\n";

// Delete the old options rows after successful migration
$wpdb->query("
    DELETE FROM {$wpdb->options}
    WHERE option_name LIKE 'formbuilder_submission_%'
");
echo "Old options rows deleted.\n";







Step 4 — Prevented future autoload abuse with an enforcement filter:

// functions.php — Prevent any option over 10KB from being autoloaded

add_filter('pre_update_option', function($value, $option, $old_value) {
    // Only intercept if this is an autoloaded option being set
    global $wpdb;
    $autoload = $wpdb->get_var($wpdb->prepare(
        "SELECT autoload FROM {$wpdb->options} WHERE option_name = %s",
        $option
    ));

    if ($autoload !== 'yes') return $value;

    $serialized_size = strlen(maybe_serialize($value));
    $max_autoload_bytes = 10 * 1024; // 10KB limit for autoloaded options

    if ($serialized_size > $max_autoload_bytes) {
        // Log the violation — this is a plugin misbehaving
        error_log(sprintf(
            'Large autoloaded option blocked: name=%s size=%dKB. ' .
            'This option should use autoload=no.',
            $option,
            round($serialized_size / 1024, 1)
        ));
        // Force autoload off for this option going forward
        $wpdb->query($wpdb->prepare(
            "UPDATE {$wpdb->options} SET autoload = 'no' WHERE option_name = %s",
            $option
        ));
    }

    return $value;
}, 10, 3);









Step 5 — Scheduled daily maintenance to keep wp_options clean:

// functions.php — Daily options table hygiene
add_action('daily_options_cleanup', function() {
    global $wpdb;

    // 1. Delete expired transients
    $wpdb->query("
        DELETE FROM {$wpdb->options}
        WHERE option_name LIKE '_transient_timeout_%'
        AND   option_value < UNIX_TIMESTAMP()
    ");
    $wpdb->query("
        DELETE FROM {$wpdb->options}
        WHERE option_name LIKE '_transient_%'
        AND   option_name NOT LIKE '_transient_timeout_%'
        AND   CONCAT('_transient_timeout_', SUBSTRING(option_name, 12))
              NOT IN (SELECT option_name FROM {$wpdb->options})
    ");

    // 2. Delete expired WC sessions
    $wpdb->query("
        DELETE FROM {$wpdb->options}
        WHERE option_name LIKE 'wc_session_%'
        AND   option_name LIKE '%_timeout'
        AND   option_value < UNIX_TIMESTAMP()
    ");

    // 3. Alert if autoloaded data exceeds 1MB
    $autoload_mb = $wpdb->get_var("
        SELECT ROUND(SUM(LENGTH(option_value)) / 1024 / 1024, 2)
        FROM {$wpdb->options}
        WHERE autoload = 'yes'
    ");

    if ($autoload_mb > 1) {
        wp_mail(
            get_option('admin_email'),
            "wp_options autoload bloat: {$autoload_mb}MB",
            "Autoloaded options are consuming {$autoload_mb}MB. Review and clean up."
        );
    }
});

if (!wp_next_scheduled('daily_options_cleanup')) {
    wp_schedule_event(time(), 'daily', 'daily_options_cleanup');
}







wp_options dropped from 180MB to 1.4MB. Page load time returned to under 1 second. The hosting provider confirmed the database connection pool was no longer saturated.

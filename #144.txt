Bug Title / Short Description (one liner) : "Every new sub-site I create gives a 404 — the main site is fine"
Bug category: Multisite Network


Description:
A digital agency managing a WordPress Multisite network for a university client reported that whenever they created a new sub-site through the Network Admin, the sub-site was listed correctly in Network → Sites but accessing it immediately returned a 404 Not Found. The main site (university.edu) worked perfectly. Sub-sites at university.edu/department-name/ were all broken. No error appeared during the sub-site creation wizard.





Main Cause:
The Multisite network was configured in subdirectory mode (university.edu/department-name/). When WordPress Multisite is installed in subdirectory mode, it requires specific rewrite rules in .htaccess (for Apache) or in the Nginx config block to correctly route sub-site requests. The server had recently been migrated to a new Apache instance, and the new server's .htaccess contained only the standard single-site WordPress rewrite rules — the Multisite-specific rewrite block was completely missing.

# What was in .htaccess — single-site rules only (MISSING multisite rules):


# BEGIN WordPress
<IfModule mod_rewrite.c>
RewriteEngine On
RewriteBase /
RewriteRule ^index\.php$ - [L]
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule . /index.php [L]
</IfModule>
# END WordPress









What We Did:




Step 1 — Confirmed the rewrite rule gap by testing sub-site URL routing:


# Test if sub-site URL is reaching WordPress at all
curl -I https://university.edu/engineering/
# HTTP/1.1 404 Not Found — Apache returning 404 before WP even loads

# Check what Apache sees for this URL
curl -v https://university.edu/engineering/ 2>&1 | grep -E "< HTTP|Server:|X-Powered"
# No WordPress headers — confirmed: request never reaches PHP/WordPress

# Verify mod_rewrite is enabled
apache2ctl -M | grep rewrite
# rewrite_module (shared) ← Good, it's enabled

# Check current .htaccess contents
cat /var/www/html/.htaccess
# Confirmed: only single-site rewrite rules present







Step 2 — Replaced the .htaccess with the correct Multisite subdirectory rules:

# .htaccess — WordPress Multisite subdirectory mode (Apache)
# Generated by WordPress Multisite setup — replaces single-site rules entirely

# BEGIN WordPress Multisite
<IfModule mod_rewrite.c>
RewriteEngine On
RewriteBase /

# Multisite — do not rewrite requests for existing files or directories
RewriteRule ^index\.php$ - [L]

# Uploaded files — route to the correct sub-site's upload directory
RewriteRule ^([_0-9a-zA-Z-]+/)?files/(.+) wp-includes/ms-files.php?file=$2 [L]

# Add a trailing slash to /wp-admin
RewriteRule ^([_0-9a-zA-Z-]+/)?wp-admin$ $1wp-admin/ [R=301,L]

# Route requests to the correct sub-site
RewriteCond %{REQUEST_FILENAME} -f [OR]
RewriteCond %{REQUEST_FILENAME} -d
RewriteRule ^ - [L]

# Everything else goes to index.php — WordPress handles the routing
RewriteRule ^([_0-9a-zA-Z-]+/)?(wp-(content|admin|includes).*) $2 [L]
RewriteRule ^([_0-9a-zA-Z-]+/)?(.*\.php)$ $2 [L]
RewriteRule . index.php [L]
</IfModule>
# END WordPress Multisite










Step 3 — For Nginx servers (documented for the agency's other deployments):

# /etc/nginx/sites-available/university.conf
# WordPress Multisite subdirectory mode — Nginx

server {
    listen 443 ssl;
    server_name university.edu;
    root /var/www/html;
    index index.php;

    # Multisite subdirectory routing
    location / {
        try_files $uri $uri/ /index.php?$args;
    }

    # Multisite uploaded files routing
    # Routes sub-site file requests through ms-files.php
    location ~ ^/[_0-9a-zA-Z-]+/files/(.+)$ {
        try_files $uri /wp-includes/ms-files.php?file=$1;
        access_log off;
        log_not_found off;
        expires max;
    }

    # Route wp-admin subdirectory access
    if (!-e $request_filename) {
        rewrite ^/[_0-9a-zA-Z-]+(/wp-.*) $1 last;
        rewrite ^/[_0-9a-zA-Z-]+(/.*\.php) $1 last;
    }

    location ~ \.php$ {
        include        fastcgi_params;
        fastcgi_pass   unix:/run/php/php8.1-fpm.sock;
        fastcgi_param  SCRIPT_FILENAME $document_root$fastcgi_script_name;
    }
}










Step 4 — Verified wp-config.php had all required Multisite constants:

// wp-config.php — Required Multisite constants (must all be present)

define('MULTISITE',           true);
define('SUBDOMAIN_INSTALL',   false);           // false = subdirectory mode
define('DOMAIN_CURRENT_SITE', 'university.edu');
define('PATH_CURRENT_SITE',   '/');
define('SITE_ID_CURRENT_SITE', 1);
define('BLOG_ID_CURRENT_SITE', 1);

// Recommended additions for Multisite stability
define('WP_ALLOW_MULTISITE',  true);
define('SUNRISE',             false);           // Set true only if using domain mapping plugin

// Upload path — critical for sub-site file isolation
define('UPLOADBLOGSDIR', 'wp-content/blogs.dir');
define('UPLOADS',        'wp-content/uploads'); // Override only if needed







Step 5 — Added a network health check that tests each sub-site's reachability:

// network-health-check.php — Run via: wp eval-file network-health-check.php --url=university.edu


$sites = get_sites(['number' => 100, 'fields' => 'ids']);

echo "Network Health Check — " . count($sites) . " sites\n\n";

foreach ($sites as $site_id) {
    $site     = get_site($site_id);
    $url      = 'https://' . $site->domain . $site->path;
    $response = wp_remote_head($url, ['timeout' => 10, 'sslverify' => false]);

    if (is_wp_error($response)) {
        $status = 'ERROR: ' . $response->get_error_message();
    } else {
        $code   = wp_remote_retrieve_response_code($response);
        $status = $code === 200 ? '✅ 200 OK' : "⚠️  HTTP $code";
    }

    printf("Site %-3d | %-50s | %s\n", $site_id, $url, $status);
}







All sub-sites became accessible immediately after the .htaccess update. The agency added a post-migration checklist that includes verifying Multisite rewrite rules before handing off any network deployment.

Bug Title / Short Description (one liner) : "Our database is 4GB and WordPress is timing out — it's almost all revisions"
Bug category: Post, Page & Content Editor


Description:
A high-volume content site with 15 full-time writers had been running for 4 years with no revision limits configured. A routine database size audit revealed the wp_posts table was consuming 3.8GB — and 94% of that was post revisions. The site was experiencing regular PHP timeouts during post saves, admin searches were timing out, and the hosting bill had tripled due to database storage overage.




Main Cause:
WordPress's default behavior stores an unlimited number of revisions for every post save. With 15 writers saving drafts multiple times per day over 4 years, the database had accumulated over 1.2 million revision records. Each revision stored a full copy of the post content — including all embedded block markup — meaning a single 50KB article had 200+ full copies stored in wp_posts. Autosave records were also never being cleaned up, adding another 300,000+ rows.





What We Did:


Step 1 — Assessed the damage with a full revision audit:

# Count total revisions vs real posts

SELECT
    post_status,
    post_type,
    COUNT(*) AS count
FROM wp_posts
GROUP BY post_status, post_type
ORDER BY count DESC;


-- Results:
-- inherit  |  revision  |  1,247,832   (the problem)
-- publish  |  post      |  12,405
-- publish  |  page      |  847



# Check total storage consumed by revisions
SELECT
    COUNT(*) AS revision_count,
    ROUND(SUM(LENGTH(post_content)) / 1024 / 1024, 2) AS content_mb,
    ROUND(SUM(LENGTH(post_content) + LENGTH(post_title) +
          LENGTH(post_excerpt)) / 1024 / 1024, 2) AS total_mb
FROM wp_posts
WHERE post_type = 'revision';

-- Result: 1,247,832 revisions consuming 3,612 MB





Step 2 — Performed a safe, batched deletion to avoid locking the database:

// cleanup-revisions.php — Run via: wp eval-file cleanup-revisions.php
global $wpdb;

$batch_size    = 500;
$total_deleted = 0;
$keep_latest   = 5; // Keep the 5 most recent revisions per post


// Get all parent post IDs that have revisions
$post_ids = $wpdb->get_col("
    SELECT DISTINCT post_parent
    FROM {$wpdb->posts}
    WHERE post_type   = 'revision'
    AND   post_parent > 0
");


echo "Processing " . count($post_ids) . " posts with revisions...\n";

foreach ($post_ids as $post_id) {
    // Get all revision IDs ordered newest first
    $revision_ids = $wpdb->get_col($wpdb->prepare("
        SELECT ID FROM {$wpdb->posts}
        WHERE post_parent = %d
        AND   post_type   = 'revision'
        ORDER BY post_date DESC
    ", $post_id));


    // Keep the N most recent, delete the rest
    $to_delete = array_slice($revision_ids, $keep_latest);
    if (empty($to_delete)) continue;

    foreach (array_chunk($to_delete, $batch_size) as $batch) {
        $placeholders = implode(',', array_fill(0, count($batch), '%d'));


        // Delete associated meta first to avoid orphaned rows
        $wpdb->query($wpdb->prepare(
            "DELETE FROM {$wpdb->postmeta} WHERE post_id IN ($placeholders)",
            ...$batch
        ));
        
        
        // Then delete the revision posts themselves
        $wpdb->query($wpdb->prepare(
            "DELETE FROM {$wpdb->posts} WHERE ID IN ($placeholders)",
            ...$batch
        ));
        
        $total_deleted += count($batch);
        echo "Deleted batch — total so far: $total_deleted\n";
        usleep(50000); // 50ms pause per batch to avoid DB overload
    }
}




// Clean up all autosave records
$autosaves_deleted = $wpdb->query("
    DELETE FROM {$wpdb->posts}
    WHERE post_type   = 'revision'
    AND   post_status = 'inherit'
    AND   post_name   LIKE '%-autosave-v1'
");

echo "Autosaves deleted: $autosaves_deleted\n";
echo "Total revisions deleted: $total_deleted\n";


// Reclaim disk space after bulk deletion
$wpdb->query("OPTIMIZE TABLE {$wpdb->posts}");
$wpdb->query("OPTIMIZE TABLE {$wpdb->postmeta}");
echo "Database optimized.\n";






Step 3 — Configured revision limits going forward in wp-config.php:

// wp-config.php — Revision and autosave control

// Keep only the 10 most recent revisions per post
define('WP_POST_REVISIONS', 10);


// Increase autosave interval to reduce autosave record accumulation
define('AUTOSAVE_INTERVAL', 120); // seconds — default is 60






Step 4 — Added per-post-type revision limits for more granular control:

// functions.php — Different revision limits per post type
add_filter('wp_revisions_to_keep', function($num, $post) {
    $limits = [
        'post'          => 10,  // Blog posts — keep 10
        'page'          => 5,   // Pages change rarely — keep 5
        'case_study'    => 15,  // High-value content — keep more
        'press_release' => 3,   // Low editorial value — keep few
        'product'       => 5,   // WooCommerce products
    ];
    return $limits[$post->post_type] ?? 5; // Default: 5
}, 10, 2);




Step 5 — Scheduled recurring revision cleanup to prevent future bloat:


// functions.php — Monthly automated revision cleanup
add_action('monthly_revision_cleanup', function() {
    global $wpdb;

    // Delete revisions older than 6 months, preserving the most recent 10
    $wpdb->query("
        DELETE p FROM {$wpdb->posts} p
        WHERE p.post_type = 'revision'
        AND   p.post_date < DATE_SUB(NOW(), INTERVAL 6 MONTH)
        AND   p.ID NOT IN (
            SELECT * FROM (
                SELECT ID FROM {$wpdb->posts}
                WHERE post_type   = 'revision'
                AND   post_parent = p.post_parent
                ORDER BY post_date DESC
                LIMIT 10
            ) AS keep_revisions
        )
    ");

    // Remove expired autosave records
    $wpdb->query("
        DELETE FROM {$wpdb->posts}
        WHERE post_type   = 'revision'
        AND   post_status = 'inherit'
        AND   post_name   LIKE '%-autosave-v1'
        AND   post_date   < DATE_SUB(NOW(), INTERVAL 7 DAY)
    ");
});

if (!wp_next_scheduled('monthly_revision_cleanup')) {
    wp_schedule_event(time(), 'monthly', 'monthly_revision_cleanup');
}






Database size dropped from 3.8GB to 210MB. Post save timeouts disappeared entirely. Hosting bill reduced by 68%.

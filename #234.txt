Bug Title / Short Description (one liner) : "Our custom post type archive page always returns 404 — the single posts work fine"
Bug category: Permalinks & URL Routing


Description: 
A real estate agency client had a custom post type property registered by their theme. Individual property pages (e.g., clientsite.com/property/beachfront-villa/) loaded perfectly. But the archive page (clientsite.com/properties/) — which should list all properties — always returned a 404. The archive page had never worked since the site launched 3 months prior and the previous developer had given up debugging it.




Main Cause:
Two separate issues were found:

The custom post type was registered with 'has_archive' => false — meaning WordPress never registered a rewrite rule for the archive URL at all. The archive slug properties had nowhere to route to.
Even after fixing that, the rewrite rules in the database were stale — WordPress caches its rewrite rules in wp_options under the key rewrite_rules. The stale cache still had no entry for properties/ even after the code fix, because nobody had flushed the rewrite rules.

// The broken post type registration found in the theme:
register_post_type('property', [
    'label'       => 'Properties',
    'public'      => true,
    'has_archive' => false, // ← This is why the archive 404s — intentionally disabled
    'rewrite'     => ['slug' => 'property'],
    'supports'    => ['title', 'editor', 'thumbnail'],
]);






What We Did:


Step 1 — Confirmed the rewrite rule was absent:

// Temporary debug — dump all registered rewrite rules
add_action('init', function() {
    if (!isset($_GET['debug_rewrites']) || !current_user_can('manage_options')) return;

    global $wp_rewrite;
    $rules = $wp_rewrite->wp_rewrite_rules();

    // Check if any rule matches the archive slug
    $property_rules = array_filter(
        array_keys($rules),
        fn($pattern) => strpos($pattern, 'properties') !== false
    );

    echo '<pre>';
    if (empty($property_rules)) {
        echo "NO rewrite rules found for 'properties' — archive is not registered.\n";
    } else {
        echo "Rules matching 'properties':\n";
        print_r($property_rules);
    }
    echo '</pre>';
    exit;
});

// Visit: https://clientsite.com/?debug_rewrites=1

Step 3 — Flushed rewrite rules after the code change:

# Always flush rewrite rules after ANY post type registration change
wp rewrite flush --hard

# Verify the archive rule now exists
wp eval '
global $wp_rewrite;
$rules = $wp_rewrite->wp_rewrite_rules();
$found = array_filter(array_keys($rules), fn($p) => strpos($p, "properties") !== false);
echo empty($found) ? "Archive rule MISSING\n" : "Archive rule found: " . implode(", ", $found) . "\n";

Step 5 — Added a rewrite rule integrity checker that fires after any post type registration:

// functions.php — Verify post types with archives have registered rewrite rules

add_action('wp_loaded', function() {
    if (!current_user_can('manage_options')) return;

    global $wp_post_types, $wp_rewrite;
    $rules = get_option('rewrite_rules', []);

    foreach ($wp_post_types as $type => $obj) {
        if (!$obj->has_archive || $type === 'post') continue;

        $archive_slug = is_string($obj->has_archive) ? $obj->has_archive : $type . 's';
        $has_rule     = false;

        foreach (array_keys($rules) as $pattern) {
            if (strpos($pattern, $archive_slug) !== false) {
                $has_rule = true;
                break;
            }
        }

        if (!$has_rule) {
            error_log("Missing rewrite rule for CPT archive: '{$type}' (slug: '{$archive_slug}'). Run wp rewrite flush.");
        }
    }
});




The properties archive loaded correctly immediately after flushing rewrite rules. The agency also got a template that matched their existing design system.
